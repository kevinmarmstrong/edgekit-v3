## 1. Thesis (one paragraph, no equivocation)

edgekit is the opinionated retrofit harness for adding agentic workflows to existing web apps. It absorbs the production friction of browser-first inference (Chrome AI, WebLLM, fallback), tool-calling safety (approval gates, audit, redaction), identity/state bridging, and end-user UX (download policy, status, graceful degradation) so a developer can ship a real, governed in-app agent in an afternoon instead of a quarter. The harness is thin where it can be and substantive where it must be — measured by whether each primitive absorbs real production friction adopters would otherwise re-solve badly, not by line count. Skills, Mission Profiles, knowledge access, governance, MCP, and AG-UI live as opt-in sibling packages so the harness moves at the speed of AI SDK while adopter apps move at the speed of their business. The product is sold on outcomes (ship faster, no rewrite, governed mutations, predictable cost, future-proof against a moving stack) — not on architecture.

## 2. First-principles outcomes (what is being bought)

A buyer adopts edgekit because they want, in order of weight:

1. **A working in-app agent this week, not this quarter** — without rewriting the app behind it.
2. **No support-ticket explosion** from browser-AI variability (download states, gesture rules, WebGPU absence, mid-download UX).
3. **A defensible answer when security/compliance asks** — approvals, telemetry, audit, identity/state bridging, no model-as-auth-layer.
4. **Predictable cost** — local-first when it works, deliberate cloud escalation when it has to, observable both ways.
5. **Future-proofing against churn** — when AI SDK ships v7, when Chrome AI changes, when WebLLM model registry shifts, when MCP catalogs evolve, the app keeps running and the upgrade is one bump.
6. **Composability into their existing stack** — Next, Astro, Rails, Django, plain HTML, WordPress, internal tools.

Every architectural decision in this plan traces back to one or more of these. Decisions that don't trace back get cut.

## 3. Constitutional rules (carried forward from DESIGN.md, refined)

These are non-negotiable, enforced where possible by CI, called out in PR review where not.

1. **Use, don't build.** Orchestration, tool loops, streaming, message formatting, tool repair, AG-UI SSE parsing — use AI SDK and ecosystem libraries (`experimental_repairToolCall`, `prepareStep`, `@ag-ui/client`). Reinvention is the v1/v2/v3 failure mode and the largest single source of bugs.
2. **Friction-absorption test, not line count.** A primitive earns its place by absorbing real production friction. Line budgets are a downstream consequence, not the rubric. (Test defined in §5.)
3. **Harness ≠ skills.** The harness ships the runtime spine. Skills, Profiles, knowledge, governance, MCP, AG-UI are sibling packages that depend on the harness, never the reverse.
4. **Modularity at every layer.** Every external dependency is behind an interface (provider, knowledge source, cache, memory store, redactor, UI). Swapping is one wrapper change; the harness never needs a rewrite to absorb a new runtime.
5. **Demos are external consumers.** Demos live in their own repos consuming published packages. The maintainer experiences upgrades as an adopter does. The closed-loop validation problem in v3 ends here.
6. **Outcomes-first marketing.** The landing page leads with the working demo and the 60-second embed. Architecture lives in docs, not the hero.
7. **CI enforces the constitution.** Per-package LOC budgets, export ADR requirements, dependency budgets, doc allowlist. Discipline that isn't mechanical drifts (this is the lesson of v3).
8. **Docs are audience-segmented, not capped.** The problem in v3 wasn't doc count, it was audience confusion — adopter docs, contributor docs, maintainer process notes, and aspirational planning surrogates all sat at the same level. Every doc declares its primary audience in a header, lives in the right tree (root entry points / `/docs/adopter/` / `/docs/thesis/` / `/lab/` / `.github/` contributor), and is reachable from a clear navigation path. Planning surrogates and self-graded readiness reports go to `/lab/`; thesis content (which is part of the product story) is kept and surfaced.

## 5. The friction-absorption test (decision rubric)

Every existing or proposed primitive is evaluated against three questions:

1. **What specific production friction does this absorb?** Concrete answer, not optionality.
2. **What would the adopter get wrong without it?** Real failure mode, not theoretical.
3. **Does the friction land for v3.5 adopters today, or only at hypothetical future scale?** Today = ship now. Future = defer or sibling.

Verdicts:

- **KEEP-CORE** — passes all three; lives in `@kevinmarmstrong/edgekit`.
- **KEEP-SIBLING** — passes 1 and 2 for a cohort; ships as opt-in sibling package.
- **DEFER** — fails 3 (no current adopter friction); move to `lab/` branch.
- **CUT** — fails 1 or 2 (no real friction absorbed); delete.

This rubric replaces the line-count heuristic. Line budgets are tools the rubric uses, not the rubric itself.

## 7. Pattern-by-pattern delivery

What v3.5 ships for each adopter need, and how it future-proofs.

### 7.1 Core agent runtime

**Delivers:** One configured entry point (`createAgent`) that wires browser-first cascade, tool registration, approval gates, telemetry, identity/state, and tool repair. Wrapper around AI SDK's `streamText`/`generateText` — never replacing what AI SDK does, only assembling and pre-wiring the surfaces an in-app agent needs.

**Pattern:**
```ts
import { createAgent, chromeAI, webLLM, tool } from '@kevinmarmstrong/edgekit'

const agent = createAgent({
  model: [chromeAI(), webLLM(), openai('gpt-4o-mini')],
  downloadPolicy: 'prompt',
  tools: { searchProducts, addToCart },
  identityProvider: () => ({ id: user.id, roles: user.roles }),
  stateProvider: () => ({ route: location.pathname, summary: '...' }),
  telemetry: (event) => posthog.capture(event.type, event),
})
```

**Future-proof because:** the cascade is an array (developer controls order); providers are interface-bound (swap chromeAI for ONNX when it ships); AI SDK upgrades are absorbed by the harness, not the adopter.

### 7.2 Developer onboarding

**Delivers:** `npx create-edgekit-app my-agent` scaffolds a working starter in <60 seconds. Generates a Mission Profile (real, not placeholder), a sample tool registration, a working `<edge-chat>` page, and a working test. A "5-minute retrofit" recipe doc covers adding `<edge-chat>` to an existing Next/Astro/Rails app.

**Pattern:**
```bash
npx create-edgekit-app my-agent --template ecommerce
cd my-agent && npm install && npm run dev
# Working agent at localhost:5173 in under 2 minutes
```

**Future-proof because:** templates are versioned; CLI pins compatible package ranges; new templates ship without affecting existing apps.

### 7.3 Integrations / tool use

**Delivers:** The retrofit pattern as recipe. Wrap existing endpoints as governed tools. Classify approval. Register. Done.

**Pattern:**
```ts
const updateOrder = tool({
  description: 'Update an order status',
  inputSchema: z.object({ orderId: z.string(), status: z.enum(['shipped', 'cancelled']) }),
  execute: async ({ orderId, status }, { session }) => {
    return fetch(`/api/orders/${orderId}`, {
      method: 'PATCH',
      headers: { Authorization: `Bearer ${session.auth.token}` },
      body: JSON.stringify({ status }),
    }).then(r => r.json())
  },
  needsApproval: true, // mutation → human-in-the-loop
})
```

MCP catalogs (`@kevinmarmstrong/edgekit-mcp`) and AG-UI backends (`@kevinmarmstrong/edgekit-agui`) are opt-in installs that surface the same `tool` shape.

**Future-proof because:** tools are the AI SDK's native shape; the host app owns execution; new tool sources (MCP, AG-UI, custom) plug into the same registration surface.

### 7.4 Upgrade paths

**Delivers:**
- Semver: minor releases never break Mission Profiles; majors ship a one-page UPGRADE recipe.
- Adapter interfaces for browser providers — Chrome AI changes → one wrapper file changes.
- Skills + Profiles boundary — the harness can re-architect internally without breaking Profile contracts.
- CHANGELOG with concrete migration steps for every notable change.

**Pattern:** adopter runs `npm update @kevinmarmstrong/edgekit`, reads CHANGELOG, applies any documented migration (usually none for minors), continues.

**Future-proof because:** the layering exists specifically to absorb upstream churn. AI SDK v7? Harness ships v0.x+1 with a migration recipe; adopter bumps one dep.

### 7.5 Component modularity

**Delivers:** Every external concern is behind an interface.

| Interface | Replaceable component | Default |
|---|---|---|
| `LanguageModel` (AI SDK) | Model provider | `chromeAI()`, `webLLM()` |
| `EdgeKnowledgeSource` | RAG backend | Markdown; swap for Qdrant, Neo4j, LangChain, custom |
| `EdgeMemoryStore` | Memory | Markdown; swap for IndexedDB, vector, server |
| `EdgeRedactor` | PII handling | none; swap for custom or `@kevinmarmstrong/edgekit-governance` |
| `EdgeAuditTrail` | Audit | none; swap for hash-chain or compliance backend |
| UI package | Render surface | `<edge-chat>` Lit component; swap for React, Vue (future) |

**Pattern:** adopter implements an interface, passes the implementation in `createAgent` config. No core changes.

**Future-proof because:** when ONNX Web Runtime ships, add a new provider wrapper. When the next-best Generative UI standard appears, add a renderer. The interface boundaries are the swap points.

### 7.6 Tech stack coverage

**Delivers:** Web component runs anywhere HTML runs. React wrapper for React-first shops. Server-side composability via AG-UI bridge. Edge runtime example via Cloudflare Pages.

**Coverage matrix:**

| Stack | Path | Sample integration |
|---|---|---|
| Next.js / Remix | `<edge-chat>` in client component or React wrapper | Documented |
| Astro | `<edge-chat>` directly | Documented |
| Vue / Svelte / Solid | `<edge-chat>` (web component is framework-agnostic) | Documented |
| Rails / Django / Laravel | `<edge-chat>` in template, `tool()` calls hit existing controllers | Documented |
| WordPress | `<edge-chat>` snippet + REST API tools | Documented |
| Plain HTML / static site | Two-line embed | Documented |
| Edge runtime (Cloudflare Workers) | AG-UI backend → `<edge-chat>` | Demo repo |

**Future-proof because:** Lit web components are a W3C standard; the harness has zero framework opinions; adapter packages (React, future Vue/Svelte) add idiomatic ergonomics without forking the core.

### 7.7 End-user UX (cascade, downloads, approvals)

**Delivers:** The spine that absorbs browser-AI variability so end users never see `NotAllowedError`, never see a WebGPU stack trace, and always see a coherent message that maps to a next step.

**Pattern (developer-side):**
```ts
createAgent({
  model: [chromeAI(), webLLM()],
  downloadPolicy: 'prompt',
  onModelStatus: ({ status, progress, message }) => {
    if (status === 'downloading') return `Loading AI… ${Math.round(progress * 100)}%`
    return message
  },
  onDownloadPrompt: async ({ provider, modelSize }) => {
    return await myConsentModal.confirm(`Download ${modelSize}?`)
  },
  onNoModel: () => 'AI is not available in this browser. Basic search is still active.',
})
```

**Pattern (end-user-side):**
- Chrome AI ready → instant agent, no prompt.
- Chrome AI downloading → progress bar, no prompt.
- Chrome AI downloadable → user-gesture-triggered consent flow with developer-controlled copy.
- WebGPU available, no Chrome AI → download prompt or auto-download per policy.
- Nothing → graceful "basic mode" message with optional cloud fallback the developer chose.

**Future-proof because:** new browser AI surfaces (Edge AI, future Safari ML) plug into the cascade as additional providers. The end-user UX contract doesn't change.

### 7.8 Agents-as-workers outcomes

**Delivers:** The behavior contract that makes agents safe to ship inside a real product.

- `registerActions` — host-owned CTAs derived from tool outputs. Validates user clicks against the tool schema. Doesn't run hidden tools from arbitrary forms.
- `needsApproval` — mutations pause until user confirms. Approval state persists across resume.
- Telemetry events — every tool call, approval decision, view render, error fires a typed event the adopter routes to their stack.
- Audit (sibling package) — hash-chained record of who decided what, when. Bring-your-own-signing for strict compliance.
- Generative UI — agent responses can include tables, charts, forms, action cards via `EdgeViewNode`. The renderer is in `@kevinmarmstrong/edgekit-ui`.
- Skills + Mission Profiles (sibling package) — agent behavior as reviewable, versionable artifacts. The path from "agent in our app" to "agent we can audit and improve."

**Future-proof because:** the contract is interface-shaped. New action types, new EdgeView nodes, new Skill metadata fields evolve without breaking the runtime.

### 7.9 Skills + Mission Profiles — structural move, not narrative demotion

Moving Skills + Mission Profiles into `@kevinmarmstrong/edgekit-skills` is a packaging decision. It is **not** a positioning decision. In all adopter-facing material — landing page, README, recipes, demos — Skills + Profiles remain the named answer to:

- "How does my team review and version agent behavior?"
- "How do I upgrade the harness without rewriting my agent?"
- "How do I compose agent capabilities the way I compose UI components?"

The sibling package install (`npm i @kevinmarmstrong/edgekit-skills`) is a one-line addition that lives in every recommended starter. The package boundary keeps the harness thin; the value story keeps Skills first-class. Marketing copy must not lead with "the lean core" in a way that implies Skills are secondary — they are first-class governance and composability primitives that happen to ship as an opt-in install for users who don't need them.

## 9. CI-enforced constraints (the constitution as code)

These run in GitHub Actions on every PR. Failing any one blocks merge. **The primary constraint is the ADR-per-runtime-capability-export rule; LOC budgets are secondary backstops that catch drift between ADR reviews.**

### Primary constraint (the real law)

- **ADR-per-runtime-capability-export.** Every public runtime capability export in every package has a one-paragraph ADR (`docs/adrs/<package>/<export-name>.md`) that answers the friction-absorption test: what production friction does this absorb, what would the adopter get wrong without it, why does the friction land today (not at hypothetical future scale). Simple public types, schema types, event payload interfaces, and compatibility re-exports can be covered by package API docs instead of one ADR per symbol. Adding a new runtime capability export without an ADR fails CI. Removing or renaming a runtime capability export requires the ADR to be updated.

### Secondary constraints (backstops + drift detection)

- **Per-package LOC budgets** (`scripts/check-budgets.mjs`) — these are *backstops* that catch drift between ADR reviews. A package can exceed budget if every export has a green ADR, but excess triggers a "review this package's ADRs" warning on the next PR.
  - `packages/core` source: ≤1,800
  - `packages/skills` source: ≤600
  - `packages/knowledge` source: ≤800
  - `packages/governance` source: ≤900
  - `packages/mcp` source: ≤400
  - `packages/agui` source: ≤500
  - `packages/ui` source: ≤1,400
  - `packages/react` source: ≤300
  - `packages/cli` source: ≤700
- **Dependency budget** — siblings may depend on the harness and one external library at most (e.g., `@ag-ui/client`). Cross-sibling dependencies require an ADR.
- **No hand-rolled-around-AI-SDK detector** — a lint rule that flags new `messages.push`, `as unknown as ModelMessage`, hand-rolled `while (true)` loops over `aiStreamText`. ADR required to override.
- **Doc audience header required.** Every markdown file under root + `/docs/` must start with `Audience: <adopter | contributor | maintainer | thesis>`. Missing or malformed header fails the build. This is the audience-segmentation enforcement (replacing the prior "≤6 docs" rule).
- **Doc tree placement** — `Audience: adopter` docs live in root or `/docs/adopter/`; `Audience: thesis` in `/docs/thesis/`; `Audience: contributor` in `.github/` or `CONTRIBUTING`-prefixed; `Audience: maintainer` in `/lab/`. Misplaced docs fail the build.
- **Demo isolation** — `packages/*/src/**` may not import from `site/**` or `examples/**`. Reverse direction is the only allowed flow. Enforced by `node scripts/check-demo-isolation.mjs`.
- **Real-package smoke** — CI runs `npm pack` on every release-shaped commit and installs into a scratch fresh app; failure blocks release. This is the durable replacement for "research:suite as ship-ready proof."

## 10. Success criteria (outcome-based)

For each adopter need, what success looks like — measurable, not architectural.

### 10.1 Time-to-value
- [ ] **A cold developer goes from `npm i` to "agent answering inside my page" in ≤10 minutes** following the quickstart, with no out-of-band help. Validated by external testers.

### 10.2 Production friction absorption
- [ ] **The flagship ecommerce demo runs cleanly across the provider matrix** (Chrome AI ready, Chrome AI downloading, WebLLM auto-download, server fallback, no-model). No `NotAllowedError`, no `WebGPU` stack trace, no broken empty chat at any state.

### 10.3 Upgrade safety
- [ ] **At least one major-version bump completes against the published demo repos with documented one-page migration.** This proves the upgrade contract works.

### 10.4 Modularity
- [ ] **One alternative for each interface ships as a docs example** — alternative knowledge source (e.g., Qdrant), alternative memory store (e.g., IndexedDB), alternative provider (cloud-only). Proves the interfaces aren't theoretical.

### 10.5 Tech-stack coverage
- [ ] **`<edge-chat>` runs verified in Next, Astro, plain HTML, and at least one server-rendered stack (Rails or Django).** Verified by a docs page per stack.

### 10.6 End-user UX
- [ ] **End-users never see a technical error.** Validated by running the demos with WebGPU disabled, Chrome AI unavailable, and offline — all produce coherent human-readable states.

### 10.7 Agents-as-workers
- [ ] **One end-to-end demo shows the full contract** — agent reads context, calls read tools, proposes a mutation, pauses for approval, executes through the host's API, emits telemetry + audit event, surfaces a confirmation with an action card for next step.

### 10.8 Marketing conversion
- [ ] **A first-time visitor identifies what edgekit is and what to do next within 10 seconds.** Validated by 5-person hallway test.

### 10.9 Distribution proof
- [ ] **`npm pack` + fresh-app install smoke is green for every package.** Demos consume published packages, not workspace aliases.

### 10.10 Bus-factor reduction
- [ ] **At least one external contributor opens a PR** (driven by clarity of CONTRIBUTING.md + small, well-bounded sibling packages that are individually reviewable).

## 11. Risks and mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Scope creep re-emerges during the refactor itself | High | High | CI-enforced budgets from Phase A onward; every new export requires an ADR |
| Sibling extraction breaks existing demos | Medium | Medium | Demos are being externalized anyway in Phase C; treat any internal breakage as adopter-friction signal |
| AI SDK v7 lands mid-refactor | Medium | Medium | Pin compatible range; ship v3.5 against v6 first; ship v0.3.1 against v7 with migration recipe |
| Marketing rewrite vs flagship demo build race | Medium | Low | Run D in parallel with C; the demo gates the marketing copy, not the reverse |
| Solo maintainer burnout from a 4-week refactor | Medium | High | Each phase has an independent exit and ships value alone. If phase F slips, phases A–E still produce a usable v0.2.5 |
| External demos diverge from harness intent | Low | Medium | Treat demo repos as the source of truth for adopter experience; harness PRs cite demo issues as motivation |
| Browser AI landscape shifts harder than expected (Chrome AI deprecated, etc.) | Low | High | Provider interface absorbs this; harness reduces to "cascade + UX + governance"; the bet survives one provider going away |
| "Thin harness" perception drifts back to "we need this primitive in core" | High | Medium | Friction-absorption test is the rubric; CI enforces the budgets; ADR requirement makes drift visible |

## 13. The honest one-paragraph close

This plan is a refactor, not a rewrite. It preserves every primitive that earns its keep on the friction-absorption test — the cascade controller, approval flows, identity/state bridging, the action-card pattern, the governance and knowledge surfaces, the Skills + Profiles abstraction — and moves them into the structural shape that makes the thesis ("thin harness, fat skills, governed retrofit") actually true rather than aspirational. The CI constraints exist because v3 proved that intent alone doesn't survive contact with new ideas. Demos move external because the closed-loop validation problem can't be fixed any other way. The marketing tightens because the demo carries the value, not the prose. Four weeks of focused work gets to v0.3.0 with a published package surface, three working external demos, a credible provider matrix, and an outside developer who installed it without help. That last metric is the only one that matters; everything else in this plan exists to make it achievable.
