local-first, privacy-first agent runtime

Add an agent to your app without handing every interaction to a cloud meter.

edgekit is open-source infrastructure for developers who want agent workflows inside real product surfaces. Run Chrome AI, WebLLM, and app-tuned local models first; call your existing APIs as tools; keep sensitive context close to the user; and route to cloud workers only when a workflow actually needs it.

Problem to solution

You want an agent in your app. These are the blockers edgekit is built to remove.

Developers do not evaluate open-source agent infrastructure like a SaaS checkout page. They arrive with a constraint. Scan the matrix for the problem that is slowing your rollout, then jump into the docs for the implementation surface.

Problem

Token costs become an open-ended liability.

Public agents can turn stuck users, power users, or scripted loops into metered cloud spend before the feature proves ROI.

edgekit

Run browser-native models first and use explicit cloud fallback only for workflows that justify the cost.

Problem

Sensitive app context crosses the wrong boundary.

Agent prompts often need identity, page state, account data, and tool results that should not automatically leave the application.

edgekit

Keep inference, memory, state summaries, and redaction local by default while the host app controls every cloud route.

Problem

The agent ignores the app you already built.

Rebuilding product logic around a new agent stack creates drift from the APIs, permissions, and workflows users already trust.

edgekit

Register existing app functions and MCP catalogs as typed tools, with the host app staying responsible for execution.

Problem

One model does not fit every workflow.

Basic navigation, product help, support triage, and deep synthesis have different latency, cost, and reasoning requirements.

edgekit

Configure Chrome AI, WebLLM, local ladders, supervisor routing, and cloud workers per use case instead of per project.

Problem

Slow orchestration makes the product feel broken.

Sequential tool calls, repeated prompts, and blank spinners make agent workflows feel heavier than the app around them.

edgekit

Use parallel-safe read tools, semantic response caching, and streaming activity states to keep the UI moving.

Problem

Network drops break cloud-only agents.

Field teams, travel workflows, and unreliable mobile connections cannot depend on a remote model for every step.

edgekit

Pair local inference with Markdown memory, offline mutation journals, and adapter contracts for CRDT-backed sync.

Problem

Autonomous mutations need trust and evidence.

Adding to cart, updating plans, suspending accounts, or touching regulated workflows cannot happen as invisible model side effects.

edgekit

Gate risky tools with approvals, emit telemetry, and record hash-chained audit trails around important actions.

Problem

Dynamic tools expand the attack surface.

Third-party MCP catalogs and generated tool calls need policy boundaries before they touch app state or credentialed systems.

edgekit

Apply RBAC manifests, identity-aware tool hydration, payload limits, timeouts, and guarded execution before broader isolation adapters.

First principles

The sidecar assists the workflow. Your app remains the authority.

edgekit is not asking you to move product logic into a chatbot. The agent gets a typed, governed view of what the app already knows how to do: identity, state, tools, approval, audit, and recovery.

01

Mount the sidecar

Add the edge-chat web component or use `createAgent()` when you need a custom interface.

02

Expose real actions

Wrap existing app functions as AI SDK tools with typed inputs and outputs.

03

Approve changes

Mark irreversible tools as guarded so the user approves before state changes.
